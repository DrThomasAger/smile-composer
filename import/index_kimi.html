The Philosophy: Why "Smiling" Makes Better Prompts

The philosophy behind (: Smile is rooted in the idea that the structure and tone of a prompt can significantly influence the quality and consistency of a model's response. The name "Smile" itself reflects this philosophy, evoking positivity, clarity, and approachability.

Just as a smile improves human interaction, a "smiling" prompt improves human-AI interaction. The use of positive text emojis as delimiters is a deliberate design decision aimed at creating a more pleasant and engaging experience for the prompt engineer.

(: Smile Syntax Guide

Understanding Sections

A "section" is a meaningfully different part of a prompt from another part. Sections help models understand context and intent more clearly.

Starting a Section

(: Section Name (

Begin with `(:`, add descriptive name, end with `(`

Closing a Section

) End Section Name :)

Use `)` or `:)` to close. Naming the end improves readability

Understanding "Eyes"

[= Straight Eyes - Strict Instructions

For rigid instructions that must be followed exactly

[= Write this word for word ["Thinking through step by step..."] then reply =]

[" Quote Eyes - Verbatim Text

Text that must be repeated word for word

Repeat back verbatim ["I will provide an accurate, honest rewrite..."]

[$ Cash Eyes - Variables

Placeholders to be replaced with actual values

Next is user input (: User input ( [$User_Input_Document$] ) End input :)

[! Important Eyes - Emphasis

Critical instructions needing special attention

[! NEVER use an emdash! !]

[; Winky Eyes - Comments

Human notes not intended for model interpretation

[; Meta-Note: User intends to improve downstream task intelligence ;]

Complete Syntax Map

Symbol Purpose Example When to Use
`(: Section (` Begin a named section `(: Format (` Starting any section including a new prompt
`)` Shortened close for current section `) End section :)` Ending a section of the prompt
`:)` Close entire Smile block `) End section :)` Final ending marker
`[: alternate section [` More rigid logical section `[: reply in Markdown [` Creating meaningful contrast between section types
`[= literal =]` Very strict instructions `[= Write this word for word... =]` Rigid instructions that must be followed exactly

A Simple Rule: More or Less Structure?

Rule: More (: Smile structure if it increases instruction following. Less structure if it increases instruction following. Experiment to find the optimal balance for your model and task.

Frequently Asked Questions

Why structure small prompts?

Models follow structured instructions more consistently. Even small prompts benefit from clear structure to prevent ambiguity and ensure intent is understood.

Why structure large prompts?

Structure enables models to follow more instructions over more turns with more agents, without context bleed or hallucination drift. It separates concerns effectively.

Why use (: Smile instead of Markdown?

(: Smile was made to be performant for LLMs. Markdown is a document format for rendering. (: Smile focuses solely on instruction-following without presentation complexity.

Is (: Smile strict or flexible?

Both. It provides a clear framework but allows flexibility in implementation. The key is finding the right balance for your specific model and task requirements.
